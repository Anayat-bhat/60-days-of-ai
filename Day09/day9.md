# Day 9 – MVP First & Iterative Development

## Objective

Today's goal was to understand why experienced AI builders develop applications in small, manageable steps instead of trying to create a complete product in a single prompt. I built an MVP version of **NutriScope** and then enhanced it by adding more features through a second prompt.

---

## What I Did Today

I started by asking Claude to build the first version of NutriScope as a simple but functional nutrition tracking application. After testing the MVP, I used a second prompt to improve the application by adding more advanced features such as CSV upload, additional food items, meal planning, risk analysis, and improved charts.

This activity helped me understand how AI can be used as a development partner through an iterative workflow.

---

## What is an MVP (Minimum Viable Product)?

A **Minimum Viable Product (MVP)** is the simplest working version of an application that includes only the essential features needed to solve a problem. Instead of spending time building every possible feature, developers first create a functional product that can be tested, improved, and expanded later.

The purpose of an MVP is to validate ideas quickly, identify improvements, and reduce development time before investing in advanced functionality.

---

## Why is Iterative Development Important?

Iterative development means improving a product step by step instead of trying to build everything at once.

This approach allows developers to:

* Build a working application faster.
* Test each version before adding new features.
* Identify problems early.
* Improve quality through continuous updates.
* Reduce errors and make development more manageable.

Today's exercise demonstrated that AI also performs better when applications are built in multiple stages rather than through one extremely large prompt.

---

## Features in the MVP Version

The first version of NutriScope included all the core functionality needed for a basic nutrition tracker.

Some of the main features were:

* Personal profile inputs (age, height, weight, activity level, and dietary preference)
* Food logging system
* Database containing 20 common food items
* Nutrition calculations
* Calorie and macronutrient tracking
* Micronutrient tracking
* Nutrition dashboard
* Basic charts
* Personalized food recommendations
* Responsive dark-themed interface

The MVP focused on providing a simple but complete user experience.

---

## Features Added in the Enhanced Version

The second version expanded the application by adding several advanced features.

New improvements included:

* CSV file upload
* More than 40 additional food items
* Additional micronutrients
* Two-day meal planner
* Nutritional risk analysis
* Educational disclaimer
* Nutrition reference sources
* Improved interactive charts
* Advanced dietary recommendations

These additions made the application more practical and feature-rich while keeping the original functionality intact.

## Biggest Insight

The biggest lesson I learned today is that **building a simple working product first is much more effective than trying to build a complete application in a single prompt**.

Starting with an MVP made it easier to understand the application's structure, test its functionality, and then improve it step by step without becoming overwhelmed.

---

## What I Learned

Today's activity taught me several important software development principles.

I learned that:

* AI performs better when tasks are divided into smaller stages.
* Building an MVP saves time and reduces complexity.
* Iterative development produces higher-quality applications.
* Testing each version before adding new features leads to better results.
* Prompt engineering is not only about asking good questions but also about improving applications through multiple iterations.
